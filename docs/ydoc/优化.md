## 性能优化分析报告

### 🔴 高优先级 — 影响明显的性能问题

---

**1. SQLite 每次写操作都 `save()` 全量序列化落盘**

`sqliteStore.ts` 第 247-251 行：

```typescript
save() {
  const data = this.db.export();      // 全量序列化整个 DB 到内存
  const buffer = Buffer.from(data);
  fs.writeFileSync(this.dbPath, buffer); // 同步写磁盘
}
```

每次 `set()`、`delete()` 都触发一次**完整的数据库序列化 + 同步写磁盘**。在流式输出期间，`coworkStore.ts` 里每条消息更新都调用 `save()`，意味着每隔几十毫秒就有一次全量磁盘写入。

> **优化方向**：引入 write debounce（如 500ms 内合并多次写入），或改为 WAL 模式 + 异步落盘。

---

**2. `coworkStore.getConfig()` 每次都发出 8 条独立 SQL 查询**

`coworkStore.ts` 第 939-957 行：

```typescript
const workingDirRow       = this.getOne('SELECT value FROM cowork_config WHERE key = ?', ['workingDirectory']);
const agentEngineRow      = this.getOne('SELECT value FROM cowork_config WHERE key = ?', ['agentEngine']);
const memoryEnabledRow    = this.getOne('SELECT value FROM cowork_config WHERE key = ?', ['memoryEnabled']);
// ... 共 8 次独立查询
```

这个函数在 `openclawConfigSync.sync()` 每次触发时都执行，而 sync 在每次配置变更时被调用。

> **优化方向**：改为 `SELECT * FROM cowork_config` 一次性取全部行，再在内存中组装。

---

**3. `CoworkSessionDetail` 消息列表无虚拟化**

`CoworkSessionDetail.tsx` 第 1790 行：

```typescript
const displayItems = useMemo(() => messages ? buildDisplayItems(messages) : [], [messages]);
```

所有消息一次性渲染进 DOM，长会话（100+ 条消息）每次更新都全量 re-render，且没有使用虚拟滚动（`react-window` / `react-virtual`）。流式输出期间每几十毫秒一次 state 更新触发整个列表重渲染。

> **优化方向**：用 `react-window` 或 `@tanstack/react-virtual` 实现虚拟列表，只渲染可视区域内的消息。

---

**4. `MarkdownContent` 在流式更新时重复创建 components 对象**

`MarkdownContent.tsx` 第 606 行：

```typescript
const components = useMemo(
  () => createMarkdownComponents(resolveLocalFilePath),
  [resolveLocalFilePath]  // resolveLocalFilePath 若每次渲染都是新引用则 memo 失效
);
const normalizedContent = useMemo(
  () => normalizeDisplayMath(encodeFileUrlsInMarkdown(content)),
  [content]
);
```

流式输出时 `content` 每次都变化，`normalizedContent` 每次都重新计算（含正则替换），如果 `resolveLocalFilePath` 父组件未 `useCallback` 稳定引用，`components` 也会每次重建。

> **优化方向**：确认父组件传入 `resolveLocalFilePath` 已用 `useCallback`（实际代码中已有，但需验证依赖项），考虑对 `normalizeDisplayMath` 做更细粒度的缓存。

---

### 🟡 中优先级 — 架构层面的效率问题

---

**5. `Settings.tsx` 过度集中导致不必要的重渲染**

`Settings.tsx` 有 **40+ 个 `useState`**，所有设置面板（通用、AI 模型、IM、记忆、Cowork 等）挤在一个巨型组件里。任何一个局部状态变化都会触发整个 Settings 组件树重渲染。

> **优化方向**：按 Tab 拆分为独立子组件（`GeneralSettings`、`ModelSettings`、`IMSettings` 等），通过 `React.lazy` 按需加载。

---

**6. `main.ts` IPC 广播用 `windows.forEach` 全量遍历**

`main.ts` 多处：

```typescript
windows.forEach((win) => {
  win.webContents.send('cowork:stream:messageUpdate', payload);
});
```

每次流式更新都遍历所有窗口广播。窗口数量增加时无开销扩展性问题，且每次 IPC 都有序列化开销。

> **优化方向**：维护一个 `activeWindowId` 只向活跃窗口发送流式事件，非活跃窗口只在 `complete` 时同步最终状态。

---

**7. `coworkStore.ts` 中 BM25 相似度计算无缓存**

`coworkStore.ts` 第 87-175 行实现了完整的 BM25 算法（含 bigram tokenization），在记忆去重和检索时被调用。每次调用都重新计算 TF-IDF 权重，没有任何缓存机制。

> **优化方向**：对 `user_memories` 的 TF-IDF 向量做增量缓存，或改用 SQLite FTS5 扩展的内置全文检索（性能提升 10x+）。

---

**8. `skillManager.ts` 读取技能文件时多次 `fs.readFileSync`**

`skillManager.ts` 第 1043-1045 行：

```typescript
const startScript        = fs.readFileSync(startServerScript, 'utf-8');
const searchScriptContent = fs.readFileSync(searchScript, 'utf-8');
const serverEntryContent  = fs.readFileSync(serverEntry, 'utf-8');
```

这三个文件在每次技能检测时都同步读取，且没有缓存。技能扫描触发频繁时（文件监听 + 变更通知）会造成不必要的磁盘 I/O。

> **优化方向**：用 `Map` 对这些静态文件内容做进程生命周期内的缓存，内容不变就不重读。

---

### 🟢 低优先级 — 小优化点

---

**9. `main.ts` 第 906 行用 `JSON.stringify` 做深比较**

```typescript
const secretEnvVarsChanged = JSON.stringify(nextSecretEnvVars) !== JSON.stringify(prevSecretEnvVars);
```

序列化两个对象再比较字符串，比直接做浅层对象比较慢，且键顺序不一致时会误判。

> **优化方向**：改用 lodash 的 `isEqual` 或手写浅比较函数。

---

**10. `openclawConfigSync.ts` 多个 `getXxxConfig()` 每次 sync 都调用**

`sync()` 方法每次都调用 8+ 个 `getXxxConfig()` 函数获取各平台配置，这些函数每次都从 SQLite 读取。频繁的 sync 触发（如多个配置连续变更）会导致重复 DB 读取。

> **优化方向**：在 `sync()` 前对 300ms 内的连续调用做 debounce（类似 imConfigSyncTimer 的模式），已有部分实现，可统一化。

---

### 汇总

| 优先级 | 问题                         | 预期收益                      |
| ------ | ---------------------------- | ----------------------------- |
| 🔴     | SQLite 每次写都全量落盘      | 流式输出期间 CPU/IO 大幅降低  |
| 🔴     | `getConfig()` 8 次独立查询 | 减少 SQL 执行次数             |
| 🔴     | 消息列表无虚拟化             | 长会话滚动流畅度显著提升      |
| 🔴     | Markdown 流式重渲染          | 输出延迟感降低                |
| 🟡     | Settings 巨型组件            | 设置页打开速度提升            |
| 🟡     | IPC 全量广播                 | 多窗口场景下网络/IPC 开销降低 |
| 🟡     | BM25 无缓存                  | 记忆检索延迟降低              |
| 🟡     | 技能文件重复读取             | 技能扫描 I/O 开销降低         |
| 🟢     | JSON.stringify 做对比        | 轻微 CPU 优化                 |
| 🟢     | ConfigSync debounce 不统一   | 减少冗余 DB 读取              |
