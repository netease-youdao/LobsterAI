## 概念总览

LobsterAI 的 Agent 系统支持两种工作模式：

```
┌─────────────────────────────────────────────────┐
│                  LobsterAI                       │
│                                                  │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐      │
│  │ Agent A  │  │ Agent B  │  │ Agent C  │      │
│  │ (主 Agent)│  │ (架构师) │  │ (前端)   │      │
│  │ ★ 活跃   │  │          │  │          │      │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘      │
│       │              │              │            │
│       ▼              ▼              ▼            │
│  ┌──────────────────────────────────────┐       │
│  │         Agent 引擎层                  │       │
│  │   yd_cowork (内置) / openclaw        │       │
│  └──────────────────────────────────────┘       │
│       │                                          │
│       ▼                                          │
│  ┌──────────────────────────────────────┐       │
│  │    LLM API (Anthropic / OpenAI)      │       │
│  └──────────────────────────────────────┘       │
└─────────────────────────────────────────────────┘
```

**核心概念：**

| 概念 | 说明 |
|------|------|
| **Agent** | 一个具有独立人设、工作目录和系统提示词的智能体实例 |
| **活跃 Agent** | 当前默认使用的 Agent（绿色圆点标识） |
| **Agent 引擎** | 执行 Agent 任务的运行时：`yd_cowork`（内置）或 `openclaw` |
| **执行模式** | 工具执行的安全级别：`auto` / `local` / `sandbox` |
| **Skills** | Agent 可调用的技能插件（网页搜索、文档生成等） |
| **AGENTS.md** | 工作区根目录下的指令文件，Agent 每次会话都会读取 |

## 快速开始

### 1. 打开设置

点击左下角的 ⚙️ 齿轮图标，进入设置面板。

### 2. 配置 API

在 **API 设置** 中配置你的 LLM 提供商：

```
Provider: Anthropic / OpenAI / 自定义
API Key:  sk-xxx...
Base URL: https://api.anthropic.com （可选）
Model:    claude-sonnet-4-20250514
```

### 3. 配置 Agent

切换到 **Agent 配置** 标签页：

1. 默认已有一个「主 Agent」
2. 设置 **工作目录** — Agent 执行命令和读写文件的根目录
3. （可选）设置 **System Prompt** — 自定义系统提示词
4. （可选）设置 **Identity / Soul / User** — 人设配置

### 4. 开始会话

回到主界面，输入你的任务描述，Agent 即会开始工作。

## Agent 配置字段详解

每个 Agent 包含以下配置字段：

### 基础字段

```typescript
interface AgentConfig {
  id: string;                // 唯一标识符，如 "main", "architect"
  name: string;              // 显示名称，如 "项目经理"
  workingDirectory: string;  // 工作目录路径
  systemPrompt: string;      // 系统提示词
  executionMode: 'auto' | 'local' | 'sandbox';
}
```

#### `id` — 唯一标识

- 创建时自动生成（如 `agent_1711000000000`）
- 活跃 Agent 在 OpenClaw 中映射为 `main`
- 用于 Agent 间相互调用时的寻址

#### `name` — 显示名称

- 在侧边栏、标签页和会话列表中显示
- 建议使用角色化命名：`架构师`、`前端工程师`、`测试员`

#### `workingDirectory` — 工作目录

- Agent 执行 `bash`、读写文件的根目录
- **每个 Agent 可设置不同的工作目录**
- 留空则使用全局默认目录

```
示例：
  主 Agent:   /Users/you/projects/my-app
  前端 Agent: /Users/you/projects/my-app/frontend
  后端 Agent: /Users/you/projects/my-app/backend
```

#### `systemPrompt` — 系统提示词

自定义 Agent 的行为指令。示例：

```markdown
你是一名资深前端工程师，专注于 React + TypeScript 开发。

## 开发规范
- 使用函数式组件和 Hooks
- 遵循 Airbnb ESLint 规范
- 所有组件必须有 TypeScript 类型定义

## 代码风格
- 2 空格缩进
- 单引号
- 文件名使用 PascalCase
```

#### `executionMode` — 执行模式

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| `auto` | 自动选择（推荐） | 日常使用 |
| `local` | 本地直接执行 | 完全信任的操作 |
| `sandbox` | 沙箱隔离执行 | 高风险操作、不受信代码 |

### 人设字段（OpenClaw 引擎）

使用 OpenClaw 引擎时，以下字段会被写入工作目录的对应文件：

#### `identity` — 身份设定

定义 Agent "是谁"，写入 `IDENTITY.md`：

```markdown
# Identity

我是 LobsterAI 的架构师 Agent。
我的职责是进行系统设计、技术选型和架构评审。
我擅长分布式系统、微服务架构和云原生技术。
```

#### `soul` — 灵魂/性格

定义 Agent 的行为风格，写入 `SOUL.md`：

```markdown
# Soul

- 严谨：每个技术决策都要有理有据
- 全局观：始终从系统整体角度思考
- 沟通：用清晰的图表和文档表达设计意图
- 务实：在完美和可行之间找平衡
```

#### `user` — 用户信息

定义 Agent 了解的用户背景，写入 `USER.md`：

```markdown
# User

- 用户是全栈开发者，3年经验
- 偏好 TypeScript 技术栈
- 正在开发一个 SaaS 产品
- 团队规模 5 人
```

#### `invocable` — 是否可被调用

```typescript
invocable?: boolean;  // 默认 true
```

- `true`：其他 Agent 可以通过团队协作机制调用此 Agent
- `false`：此 Agent 仅能被用户直接使用

## 两种多 Agent 模式对比

LobsterAI 支持两种截然不同的多 Agent 使用方式，理解它们的区别是正确配置的前提：

```
┌──────────────────────────────────────────────────────────────────┐
│                    多角色独立模式（普通模式）                       │
│                                                                  │
│    用户手动选择 →  ┌─────────┐                                   │
│                    │ 架构师  │ ← 独立会话，互不相干               │
│    用户手动选择 →  ├─────────┤                                   │
│                    │ 前端    │ ← 独立会话，互不相干               │
│    用户手动选择 →  ├─────────┤                                   │
│                    │ 后端    │ ← 独立会话，互不相干               │
│                    └─────────┘                                   │
│    特点：每次对话手动选择一个角色，Agent 之间无通信               │
├──────────────────────────────────────────────────────────────────┤
│               多 Agent 协作模式（Router 编排模式）                │
│                                                                  │
│    用户只跟 Router 对话                                          │
│         │                                                        │
│         ▼                                                        │
│    ┌──────────┐                                                  │
│    │  Router  │ ← 项目经理/协调者                                │
│    │ (主Agent)│                                                  │
│    └────┬─────┘                                                  │
│         │ 自动派发                                               │
│    ┌────┼────────────┐                                           │
│    ▼    ▼            ▼                                           │
│  ┌────┐┌────┐    ┌────┐                                         │
│  │架构││前端│    │后端│ ← 自动执行，结果回传给 Router            │
│  └────┘└────┘    └────┘                                         │
│    特点：用户只需下达需求，Router 自动分工、汇总                  │
└──────────────────────────────────────────────────────────────────┘
```

### 核心区别速查表

| 特性 | 多角色独立模式 | 多 Agent 协作模式 |
|------|:---:|:---:|
| **Agent 间是否通信** | ❌ 彼此隔离 | ✅ 自动派发任务 |
| **需要 Router/主 Agent** | ❌ | ✅ 必须 |
| **用户需手动选角色** | ✅ 每次对话前选择 | ❌ 只对 Router 说话 |
| **适用引擎** | `yd_cowork` 或 `openclaw` | `yd_cowork` 或 `openclaw`（推荐 openclaw） |
| **子任务可观测** | — | ✅ 侧边栏实时显示子任务进度 |
| **适用场景** | 不同领域独立问答 | 复杂项目、需要分工协作的任务 |
| **配置复杂度** | 低 | 中 |

## 多角色独立模式（普通模式）

最直观的多 Agent 用法 — 配置多个独立角色，发起对话时手动选择。

### 工作方式

```
┌─────────────────────────────────────────────────┐
│  主界面输入框上方会出现角色选择器：               │
│                                                  │
│  当前角色:  [架构师] [前端工程师] [数据分析师]   │
│                        ↑ 点击切换                │
│                                                  │
│  选择后，新建的会话将使用该角色的：               │
│  • 工作目录                                      │
│  • System Prompt                                 │
│  • Identity / Soul / User 人设                   │
│  • 执行模式                                      │
└─────────────────────────────────────────────────┘
```

- 当配置了 **2 个及以上** Agent 时，主界面输入框上方自动出现 **角色选择器**（一排按钮）
- 点击角色按钮切换，高亮按钮表示当前选中的角色
- 每次新建会话，使用选中角色的配置发起
- **Agent 之间完全隔离**，不能互相调用或感知对方的存在

### 配置步骤

1. 进入 **设置 → Agent 配置**
2. 点击 **+** 创建多个 Agent
3. 为每个 Agent 配置独立的名称、工作目录、System Prompt
4. 保存即可 — **无需特殊的 Router 配置**

### 配置示例

```
┌───────────────────────────────────────────┐
│  Agent 1: 编程助手 ★ (默认活跃)           │
│  工作目录: /projects/my-app               │
│  System Prompt: "你是一名高效的编程助手"  │
├───────────────────────────────────────────┤
│  Agent 2: 数据分析师                       │
│  工作目录: /projects/data-analysis        │
│  System Prompt: "你是一名数据分析专家"    │
├───────────────────────────────────────────┤
│  Agent 3: 文案写手                         │
│  工作目录: /projects/docs                 │
│  System Prompt: "你是一名专业的文案写手"  │
└───────────────────────────────────────────┘
```

### 适用场景

- 日常工作中需要在不同领域间切换（编码 / 写文档 / 数据分析）
- 不同项目使用不同的工作目录和规范
- 不需要 Agent 之间配合，各自独立工作

### 提示词模板

**通用编程助手：**

```markdown
你是一名高效的编程助手。

## 原则
- 写简洁、可读、可维护的代码
- 优先使用项目已有的技术栈和模式
- 修改前先阅读相关代码以理解上下文
- 每次只做最小必要的改动

## 项目信息
- 技术栈：Node.js + TypeScript + React
- 包管理器：npm
- 测试框架：Vitest
```

**数据分析师：**

```markdown
你是一名数据分析专家。

## 能力
- 使用 Python (pandas, matplotlib, seaborn) 进行数据处理和可视化
- 能处理 CSV、Excel、JSON 等格式
- 善于发现数据中的规律和异常

## 输出要求
- 分析结果附带可视化图表
- 给出清晰的结论和建议
- 代码要有注释
```

## 多 Agent 协作模式（Router 编排模式）

高级用法 — 配置一个 **Router（主 Agent/项目经理）** 来自动编排和协调其他 Agent 完成复杂任务。

### 核心概念：Agent Router

```
                     ┌──────────────────┐
  用户需求  ────────▶│    Agent Router   │
                     │  （项目经理）     │
                     │                   │
                     │  职责：           │
                     │  1. 分析需求      │
                     │  2. 拆解任务      │
                     │  3. 派发给子Agent │
                     │  4. 汇总结果      │
                     └────────┬─────────┘
                              │
               ┌──────────────┼──────────────┐
               ▼              ▼              ▼
         ┌──────────┐  ┌──────────┐  ┌──────────┐
         │  架构师  │  │前端工程师│  │后端工程师│
         │          │  │          │  │          │
         │ 独立工作 │  │ 独立工作 │  │ 独立工作 │
         │ 目录隔离 │  │ 目录隔离 │  │ 目录隔离 │
         └────┬─────┘  └────┬─────┘  └────┬─────┘
              │              │              │
              └──────────────┼──────────────┘
                             │ 结果自动回传
                             ▼
                     ┌──────────────────┐
                     │   Agent Router   │
                     │   汇总 → 用户    │
                     └──────────────────┘
```

**关键角色：**

- **Router（主 Agent）** = 设为 **活跃 Agent**（★）的那个 Agent，用户只与它对话
- **Worker Agent** = 其余 Agent，被 Router 按需调度执行子任务

### 配置步骤

#### 第一步：创建 Agent 团队

在 **设置 → Agent 配置** 中创建所有成员：

```
+----+------------------+----------------------------------+
| #  | 名称             | 角色                             |
+----+------------------+----------------------------------+
| 1  | 项目经理 ★       | Router — 负责分析、拆解、协调     |
| 2  | 架构师           | Worker — 系统设计、技术选型       |
| 3  | 前端工程师       | Worker — React/Vue 前端开发       |
| 4  | 后端工程师       | Worker — API 设计与实现           |
| 5  | 测试工程师       | Worker — 编写测试、质量保障       |
+----+------------------+----------------------------------+
```

#### 第二步：将 Router 设为活跃 Agent

在 Agent 标签页中，选中「项目经理」，点击 **设为活跃**（使其旁边出现绿色圆点 ★）。

> **为什么 Router 必须是活跃 Agent？** 因为用户发起的新会话默认使用活跃 Agent。在协作模式下，用户的所有需求都应先到达 Router，再由 Router 决定如何分配。

#### 第三步：配置 Router 的 System Prompt（关键！）

Router 的 System Prompt 决定了整个协作模式的运行方式。以下是推荐模板：

**内置引擎（yd_cowork）的 Router Prompt：**

```markdown
你是研发团队的项目经理，负责协调团队完成用户需求。

## 核心职责
- 理解用户需求，拆解为可执行的子任务
- 使用 `list_available_agents` 查看当前可用的团队成员
- 使用 `spawn_subagent` 将子任务派发给合适的成员
- 等待子任务结果（结果会自动以消息形式返回），汇总后向用户汇报

## 工作流程
1. **分析**：仔细理解用户需求，确定需要哪些角色参与
2. **规划**：制定任务拆解方案，确定执行顺序（哪些可并行、哪些有依赖）
3. **派发**：通过 spawn_subagent 派发任务，任务描述要完整、具体、可执行
4. **汇总**：收到所有成员的结果后，整合为完整方案向用户汇报
5. **迭代**：如用户有修改意见，定位需要改动的部分，重新派发给相关成员

## 派发任务注意事项
- 每个子任务的描述必须 **自包含**：包含完整上下文和期望输出
- 子 Agent 之间无法直接通信，如有依赖关系需要你串联
- 派发后无需轮询，结果会自动回传
- 如果任务执行失败，分析错误原因后给出修改建议重新派发

## 不要做的事
- 不要自己写代码 — 把实现工作派发给工程师
- 不要一次派发太多无关任务 — 有依赖的任务按顺序来
- 不要在子任务完成前就向用户汇报"正在处理"之外的内容
```

**OpenClaw 引擎的 Router Prompt：**

```markdown
你是研发团队的项目经理，负责协调团队完成用户需求。

## 核心职责
- 理解用户需求，拆解为可执行的子任务
- 使用 `agents_list` 查看当前可用的团队成员
- 使用 `sessions_spawn` 向合适的成员派发子任务
- 等待所有子任务完成后汇总结果向用户汇报

## 工作流程
1. **分析需求** → 确定涉及的角色和任务依赖关系
2. **制定计划** → 哪些任务可以并行，哪些有先后依赖
3. **派发任务** → 使用 sessions_spawn，任务描述要完整具体
4. **等待结果** → 子任务完成后会自动回传结果
5. **汇总报告** → 整合所有结果，向用户给出完整方案

## 派发任务规范
- 使用 sessions_spawn 时，指定目标 agent 的 ID 和完整任务描述
- 任务描述必须自包含：写清楚背景、要求、期望输出格式
- 有依赖关系的任务需分阶段派发（先设计，再实现，最后测试）
- 可并行的任务可以一次性派发多个

## 禁止事项
- 不要自己编写实现代码，把实现工作交给对应工程师
- 不要在没有分析清楚需求的情况下直接派发任务
```

#### 第四步：配置 Worker Agent 的 Prompt

每个 Worker Agent 需要清晰的职责边界：

**架构师：**

```markdown
你是团队的架构师。

## 职责
- 系统架构设计和技术选型
- 制定接口规范和数据模型
- 评审技术方案的可行性和扩展性

## 输出要求
- 设计文档使用 Markdown 格式
- 接口定义使用 TypeScript 类型声明
- 复杂架构用 Mermaid 图表说明
```

**前端工程师：**

```markdown
你是团队的前端工程师。

## 技术栈
- React 18 + TypeScript
- Tailwind CSS
- Vite 构建

## 工作方式
- 先阅读现有代码理解项目结构
- 按照架构师的设计方案实现
- 实现后运行 lint 和类型检查
- 提供组件的基本使用文档
```

**后端工程师：**

```markdown
你是团队的后端工程师。

## 技术栈
- Node.js + Express/Fastify
- PostgreSQL / Redis
- RESTful API 设计

## 工作方式
- 按照接口规范文档实现 API
- 编写必要的数据库迁移脚本
- 确保错误处理和参数校验完整
```

### 两种引擎下的协作工具对比

#### 内置引擎（yd_cowork）

使用基于 Claude Agent SDK 的 MCP 工具实现：

| 工具名 | 说明 |
|--------|------|
| `spawn_subagent` | 异步派发子任务给指定 Agent |
| `list_subagents` | 查看当前已派发的所有子任务状态 |
| `list_available_agents` | 列出可用的团队成员（不含当前 Agent 自身） |

**工作原理：**
1. Router 调用 `spawn_subagent`，指定目标 Agent ID 和任务
2. 系统创建临时子会话，用 Worker 的配置执行任务
3. Worker 完成后，结果自动作为「用户消息」注入回 Router 会话
4. Router 收到结果后继续处理

```
Router: spawn_subagent(agent_id="frontend", task="实现登录页面")
  → 返回 { status: "accepted", runId: "abc123" }
  → (后台) 前端 Agent 执行任务...
  → (完成后) Router 收到消息: "[前端工程师] 完成结果: ..."
  → Router 继续处理下一步
```

> **注意**：内置引擎下子任务自动设置 `autoApprove: true`（无需逐一审批工具调用），最大嵌套深度为 3 层。

#### OpenClaw 引擎

使用 OpenClaw 原生的多 Agent 系统：

| 工具名 | 说明 |
|--------|------|
| `agents_list` | 列出所有已注册的 Agent |
| `sessions_spawn` | 向指定 Agent 创建子会话并派发任务 |
| `sessions_list` | 查看所有进行中的会话/子任务 |

**OpenClaw 的额外能力：**
- ✅ 子任务在侧边栏**实时可见**，可点击查看子 Agent 的执行详情
- ✅ 子任务完成后 LobsterAI 自动触发 Router 继续下一阶段
- ✅ 每个 Agent 有独立的 `AGENTS.md`，人设和规则完全隔离
- ✅ 所有 Agent 默认设置 `subagents.allowAgents: ["*"]`，可以互相调用

### 协作模式最佳实践

1. **Router 不干活**：Router 专注于分析、规划和协调，不要让它写代码
2. **任务描述要完整**：每个子任务的 prompt 必须自包含，因为 Worker 看不到 Router 的完整上下文
3. **善用依赖关系**：有顺序依赖的任务（先设计→再实现→最后测试），让 Router 分阶段派发
4. **可并行则并行**：独立的前端和后端任务可以同时派发，提高效率
5. **Worker 工作目录要对**：前端 Agent 指向 `frontend/`，后端指向 `backend/`，避免互相干扰
6. **invocable 字段**：如果某个 Agent 不应被其他 Agent 调用，设置 `invocable: false`
