## 测试前置约束

- S1 与 S2 不要复用同一个会话；S2/S6/S7/S8 建议各自从新会话开始。
- S3、S4、S5 必须建立在同一条“已成功上传文件的会话链路”上。
- 如果同一会话先进行了纯文本轮次，再尝试上传文件，可能命中 `bridge_mode_locked`；当前应记录为“需要新会话后再测文件链路”。
- 若 UI 再次出现错误，请优先记录完整错误文本，不要只记录“失败”。

## 监测点

### 服务端日志

- 关注关键日志前缀：`[AgentApiServer][bridge-metrics]`
- 关注错误码：`bridge_file_missing`、`bridge_mode_locked`
- 关注 SSE 事件：`session.bound`、`state.changed`、`assistant.delta`、`assistant.final`、`permission.request`

### 客户端行为

- 发送前附件条中是否区分：
  - 本地新上传附件
  - 显式 `Re-attach` 的历史文件
- 历史文件是否只有点击 `重新附加` 后才进入当前发送
- 当前轮发送完成后，下一轮是否默认不继续带上上轮文件

### 性能字段

从 `bridge-metrics` 日志记录以下字段：

| 字段             | 说明                    |
| -------------- | --------------------- |
| mode           | `text-fast` / `agent` |
| firstStateMs   | 首个状态事件耗时              |
| ttftMs         | 首个 assistant token 耗时 |
| totalMs        | 整轮完成耗时                |
| assistantChars | 最终输出长度                |
| fileCount      | 当轮文件数                 |
| outcome        | `success` / `error`   |
| errorCode      | 失败时的错误码               |
